# Limitations, Risks, and Improvement Plan

This document proactively discloses the current limitations of the MVP. This is not a weakness; clearly explaining limitations makes the project more mature and trustworthy.

## 1. Limitations of the Silent Disaster Zone Detection API

### 1.1 Report data may currently be mock data

If the current report data is simulated, it can validate the “high-risk but low-report” logic, but it cannot be claimed to fully reflect real reporting conditions.

Improvements:

- Connect LINE Bot report data
- Connect form-based report data
- Connect local government disaster report systems
- Connect field verification results

### 1.2 A neural network trained with pseudo-labels cannot be claimed as a real disaster predictor

If the model is trained from pseudo-labels generated by rule-based scores, it mainly verifies the scoring-layer architecture. It has not yet learned real disaster patterns.

Improvements:

- Retrain with historical disaster records
- Use field verification results as labels
- Use report-delay data to evaluate whether “silence” indicates underreporting
- Add expert-labeled priority areas

### 1.3 Realtime data sources may be unstable

Rainfall, road, landslide, and other data sources may be affected by API availability, format changes, connection failures, or authorization limits.

Improvements:

- Preserve raw snapshots
- Mark data time
- Fall back to the latest successful snapshot
- Expose data source status in API responses

### 1.4 Spatial data and village boundaries may differ by version

Different datasets may use different coordinate systems, boundary versions, or update times. This may affect spatial joins.

Improvements:

- Standardize coordinate systems
- Record data versions
- List data sources and update dates in documentation
- Add preprocessing validation scripts

## 2. Limitations of the Disaster Volunteer Dispatcher API

### 2.1 AI dispatch suggestions may be unstable

LLMs may output inconsistent formats, fail to reference valid volunteer IDs, or generate suggestions that sound plausible but do not fully match the data.

Improvements:

- Add strict JSON output schema
- Use function calling or structured output
- Validate AI outputs
- Fall back to local deterministic algorithm when AI fails

### 2.2 The current local algorithm may be too simple

If fallback only considers distance or simple skill matching, it may ignore:

- Volunteer workload
- Previous assignment history
- Road accessibility
- Volunteer equipment
- Tasks requiring multiple people
- Safety risks

Improvements:

- Add required number of volunteers per task
- Add maximum task load per volunteer
- Add road accessibility
- Add skill weights
- Add task completion feedback

### 2.3 Volunteer privacy and safety risks

A dispatch system may process volunteer location, skills, and availability. Poor data handling may create privacy risks.

Improvements:

- Use de-identified IDs
- Do not pass names, phone numbers, national IDs, or addresses through the API
- Limit location precision
- Add access control
- Keep necessary but minimal logs

### 2.4 Local Ollama inference depends on hardware and model setup

The speed and stability of local LLM inference depends on hardware, model size, GPU availability, memory, and first-load time.

Improvements:

- Recommend lightweight models
- Allow AI to be disabled and use local algorithm only
- Set timeouts
- Warm up models
- Show Ollama status in the health endpoint

## 3. Integration limitations

### 3.1 Silent risk is not the same as a rescue task

The output of the Silent Disaster Zone Detection API means “needs verification.” It should not automatically become “send volunteers into a dangerous area.” A human commander or rule layer must determine task type and safety.

Improvements:

- Add task-generation rules
- Separate tasks into remote contact, field verification, road check, and supply confirmation
- For high-risk areas, suggest contacting village chiefs or fire stations first instead of immediately sending volunteers in

### 3.2 Converting GeoJSON coordinates into task coordinates requires care

A village centroid may not represent the true risk location. Some villages cover large areas, and the centroid may not be a suitable dispatch destination.

Improvements:

- Use hazard hotspot coordinates
- Use road-accessible points
- Use shelters, village offices, or fire stations as meeting points
- Mark coordinate source in each task

### 3.3 Scores must be explainable

A score alone is hard for commanders to trust. The system should provide `silent_reason`, data time, data sources, and score breakdown.

Improvements:

- Show rule-based score and model score
- Show report count, rainfall, road events, and sensor coverage gap
- Display reason tags on maps

## 4. Claims to avoid

To avoid overclaiming, the README and presentation should not use the following claims:

| Avoid saying | Say this instead |
|---|---|
| We can accurately predict disasters | We provide a priority verification list for high-risk, low-report areas |
| AI automatically decides disaster dispatch | AI provides dispatch suggestions, and humans make final decisions |
| No reports means there must be a problem | No reports plus high risk means further verification is needed |
| The system can replace government platforms | This project provides modular disaster-response components that can integrate with existing systems |
| The model is ready for production deployment | The MVP verifies data flow and core functionality; more data and validation are needed before deployment |

## 5. Roadmap

### Short term

- Complete bilingual README and docs in the main repository
- Add an integration demo script
- Add sample input / output
- Verify that both API Swagger pages open correctly
- Add a task-generator example

### Medium term

- Connect real report data
- Add road accessibility analysis
- Improve volunteer skill weighting and required personnel per task
- Convert AI output into structured JSON
- Add Docker Compose demo

### Long term

- Retrain models with historical disaster and field verification data
- Integrate with LINE Bot, Google Chat, or Web Dashboard
- Build human review interface
- Create task feedback loop
- Develop the project into a reusable disaster-response component library for civic tech teams
