# AI 治理、人工覆核與責任界線

## 系統風險

這是可能影響人員注意力與資源分配的防災決策支援系統。錯誤排序可能使人員忽略某些區域；錯誤派工可能造成志工技能錯配、重複投入或安全風險。因此，治理不是附註，而是系統設計的一部分。

## 治理原則

| 原則 | 實作要求 |
|---|---|
| 可解釋 | 顯示風險分數構成、資料模式、來源狀態、志工分數拆解與警示。 |
| 人在迴路 | 通報審核、任務建立、志工實際派遣皆由人員確認。 |
| 最小權限 | 一般查詢、管理通報、執行 pipeline、LINE webhook 應使用不同權限與密鑰。 |
| 可追溯 | 保存 `run_id`、資料時間、來源狀態、審核決定、任務轉換與派工 response。 |
| 不確定性揭露 | sample、batch、unverified、stale、來源失敗必須明確呈現。 |
| 個資最小化 | 只收集派工所需資訊；LINE 身分採 HMAC，建立保存／刪除規則。 |
| 安全失效 | AI／外部 API／地圖服務失敗時，回傳明確警示，不假裝成功。 |

## 人工覆核關卡

```text
民眾回報
  ↓
人工審核：verified / rejected
  ↓
風險或事件隊列
  ↓
人工確認：是否建立任務、任務類型、所需技能
  ↓
派工候選結果
  ↓
人工協調：是否通知、派遣、升級至專業單位
```

### 必須人工處理的情況

- `data_mode` 不是可信的 `live`，或 `freshness` 過期。
- 有 `has_source_issues=true`。
- 通報仍為 `pending`。
- `unassigned_tasks` 非空。
- 志工技能未完整匹配、ETA 過長或信心低。
- 任務涉及受困人員、醫療、危險區進入、夜間或道路不明。
- AI 文字與規則式資料不一致。

## 不允許的自動化行為

- 以風險分數宣告災害已發生。
- 在未審核下直接把民眾回報納入正式排序。
- 向志工自動發送具有強制性的救援／撤離命令。
- 以模型輸出決定撤離、封路、停班停課或生命安全。
- 將個人位置、LINE 身分或敏感回報公開於未授權介面。

## 最低稽核紀錄

每次正式任務與派工建議至少應記錄：

```text
request_id / task_id / incident_id
silent-zone run_id
data_mode / freshness / source_status
使用的 priority → urgency 映射
志工名冊版本與可用性時間
dispatch response / warnings / unassigned tasks
人工審核者、時間、決定與理由
```

## 部署治理

- 使用 HTTPS、反向代理與分層環境變數。
- 管理 API 需額外權限；不要把 key 放入前端。
- 在正式部署前做輸入驗證、限流、依賴更新與安全掃描。
- 志工資料與通報資料需要明確的保存期限、刪除流程與存取權限。
- 任何公開展示應使用去識別、樣本或取得同意的資料。
