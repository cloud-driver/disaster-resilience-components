# 資料來源、資料模式與可信度

## 資料不是越多越好，而是必須可追溯

本系統的風險排序只適合作為人工確認輔助。每一輪輸出都應讓使用者知道：資料從哪裡來、何時產生、哪些來源失敗、現在是不是 sample／batch／live。

## 沉默元件的資料類型

| 類型 | 例子 | 用途 | 注意事項 |
|---|---|---|---|
| 空間基礎資料 | 村里界圖資 | 將各項資料對應到村里。 | 邊界版本與行政區異動需可追溯。 |
| 靜態風險 | 淹水潛勢、土石流影響、人口／高齡特徵 | 建立 `static_risk_score` 與解釋原因。 | 靜態資料不能被說成即時災情。 |
| 即時訊號 | 中央氣象署雨量、土石流警戒、道路／事件資料 | 建立 `sensor_realtime_score`、`realtime_event_score`。 | 外部 API 失敗或延遲時須在 metadata 揭露。 |
| 觀測缺口 | 感測器覆蓋或可觀測性資料 | 建立 `sensor_gap_score`。 | 缺口表示不確定性增加，不是災情證據。 |
| 民眾／人工通報 | LINE、manual、API 報告 | 經人工審核後生成近時通報特徵與可信事件。 | `pending` 不能進入正式排序。 |
| 志工資料 | 技能、位置、可用性、選用 LINE 身分 | 供派工匹配。 | 屬個人資料，必須最小化、授權與保存期限管理。 |

## `data_mode` 的使用規則

| Mode | 意義 | 可以怎麼用 | 不可以怎麼說 |
|---|---|---|---|
| `live` | 同一輪即時 pipeline 已完成。 | 當輪人工確認參考。 | 不等於官方災害宣告。 |
| `batch` | 完整批次產出。 | 分析、展示、回溯。 | 不可說成即時。 |
| `sample` | repo 內建樣本。 | demo、UI／schema 驗證。 | 不可說成真實或當前災情。 |
| `unverified` | 找到輸出檔但缺可信 manifest。 | 先人工檢查。 | 不可直接用於決策。 |

同時應檢查：

- `generated_at`
- `generated_age_seconds`
- `freshness`
- `source_status`
- `has_source_issues`
- `run_id`
- `scoring_mode`
- `model_status`

## 通報資料生命週期

```text
LINE / Manual / API
        ↓
     pending
     ├── verified → 可用於 6h／24h 特徵與 verified_incident_queue
     └── rejected → 保留審核記錄，不納入正式分析
```

LINE user ID 應以 `REPORTER_HASH_SECRET` 做 HMAC 雜湊；不應保存身分證號、電話、完整住址等不必要個資。

## 志工資料治理

志工派工只應收集派工所必需的資料：

- 技能。
- 可出勤狀態。
- 足以估算距離的位置資料。
- 若需要個人通知，最小化的 LINE 身分資訊。

正式化後至少應補上：

1. 志工資料用途告知與同意。
2. 保存期限與刪除方式。
3. 管理者角色與存取記錄。
4. 資料庫加密／備份與事故通報程序。
5. 地址轉座標資料與原始地址的最小留存原則。

## 對外資料使用原則

- 使用資料前確認 API／資料集授權、頻率限制與再散布條件。
- 將原始資料抓取時間、來源 URL／識別碼與轉換版本記錄於 manifest。
- 某來源失敗時，寧可輸出「來源異常」而不是把舊資料偽裝成最新。
- 不用單一來源的缺失推論某地一定安全或一定有災。
