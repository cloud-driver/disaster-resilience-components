# 問題故事與使用情境

## 情境：災害初期的「安靜」不等於安全

在快速發展的豪雨、土石流、堰塞湖溢流或道路中斷情境中，指揮端最先收到的通常是已成功送出的通報。這很容易形成錯覺：沒有訊息的地方看起來似乎比較安全。

但「沒有通報」可能來自：

- 通訊中斷或停電。
- 道路受阻，現場無法回報或外界無法進入。
- 感測與監控覆蓋不足。
- 高齡、獨居或數位工具使用門檻較高。
- 災情仍在發展，尚未有人能確定如何描述。

因此，本作品的核心不是用演算法宣告某地「有災」，而是協助人員問出更好的問題：

> 在有限查核人力下，哪幾個高風險又缺少訊息的地點，最值得先主動確認？

## 主要角色

| 角色 | 困難 | 元件提供的協助 |
|---|---|---|
| 沉默區居民 | 可能無法發出求救或回報。 | 增加被納入人工查核優先序的機會。 |
| 地方災防／應變人員 | 資訊零散、人力有限，難決定先去何處。 | 提供可解釋的沉默風險與資料狀態。 |
| 村里／社區防災人員 | 難知道轄區哪裡是資訊死角。 | 以村里尺度列出待確認候選。 |
| 志工協調者 | 人力集中、技能錯配或重複派遣。 | 將已確認需求轉成任務，提供候選與警示。 |
| 民眾目擊者 | 回報格式零散、可信度未知。 | 透過 LINE／API 進入 `pending`，由人工審核後才影響正式分析。 |

## 從訊號到行動

```text
外部資料與通報
  ↓
沉默元件：風險、觀測缺口、近期已驗證通報
  ↓
A. silent_watch_queue：高風險但資訊不足，優先人工確認
B. verified_incident_queue：已有可信事件，優先人員研判
  ↓
人工確認「是否要建立任務、任務類型與需求」
  ↓
IntegratedTask
  ↓
志工元件：技能、距離、可用性、急迫度匹配
  ↓
人員覆核與實際協調
```

## 系統做什麼、不做什麼

| 做什麼 | 不做什麼 |
|---|---|
| 排序需要優先人工確認的候選區域。 | 判定災害已發生。 |
| 將已驗證的民眾回報與沉默區概念分開。 | 讓未驗證回報直接改變正式排名。 |
| 提供志工候選、ETA、分數拆解與警示。 | 直接向志工下達不可撤回的自動命令。 |
| 讓 AI 整理既有結果或做選用異常檢查。 | 讓 AI 決定撤離、封路、救援優先序或生命安全。 |

## 為何是兩個元件，而不是一個大平台

- 沉默區偵測可被地圖、GIS、儀表板或查報流程單獨使用。
- 志工派工可接受其他來源建立的任務，不依賴沉默區元件。
- 兩者以標準任務格式銜接，可被任一端替換或擴充。
- 這比把所有功能綁死在同一個應用中，更符合可重複使用的積木式設計。

## 成功標準

MVP 的成功不應用「預測準確率」或「必定救到人」描述，而應驗證：

1. 系統是否能明確揭露資料來源、時間與可信度。
2. 高風險、低觀測、低通報結果是否可被理解為人工確認優先序。
3. 結果能否轉成結構化任務。
4. 志工推薦是否呈現理由、技能／距離因素與警示。
5. 最終決策是否仍由人員掌握。
