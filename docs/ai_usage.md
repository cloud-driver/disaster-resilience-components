# AI 使用說明

## 1. AI 在本專案中的角色

本專案的 AI 不是用來「取代指揮官」，也不是用來直接判斷災害是否確定發生。AI 的角色是**輔助分析與輔助派遣**。

兩個元件中的 AI 使用方式不同：

| 元件 | AI / 智慧化角色 | 是否做最終決策 |
|---|---|---|
| 沉默災區偵測 API | 使用規則式分數與神經網路 scoring layer 產生沉默風險分數 | 否 |
| 救災志工智慧分配 API | 使用本地 Ollama LLM 輔助任務派遣推理，並提供本地演算法 fallback | 否 |

## 2. 沉默災區偵測 API 的 AI 使用

### 2.1 Scoring layer

此元件先建立可解釋的規則式分數，再透過神經網路 scoring layer 產生 `silent_risk_score`。

概念如下：

```text
base_risk_score
= static risk
+ sensor gap
+ realtime event signal

silence_factor
= lower report activity → higher silence factor

silent_risk_score
= high risk × low report activity
```

### 2.2 為什麼保留規則式分數？

保留 `silent_risk_rule_score` 的原因是讓評審、使用者與指揮者能看到分數邏輯，而不是只看到一個黑箱結果。

AI scoring layer 的價值在於：

- 未來可以替換成更準確的模型
- 可以用歷史災情資料重新訓練
- 可以和規則式分數比較
- 可以保留模型版本與 metadata，方便稽核

### 2.3 目前限制

MVP 階段如果使用 pseudo-label 訓練模型，不能宣稱模型已經能準確預測真實災情。比較嚴謹的說法是：

> 目前模型用來驗證 scoring layer 可替換架構，正式部署前應使用歷史災情、巡查結果、通報延遲資料與專家標註重新訓練。

## 3. 救災志工智慧分配 API 的 AI 使用

### 3.1 本地 Ollama 推理

Dispatcher API 使用本地 Ollama 執行 LLM。這種設計的優點是：

- 可在本地或內部環境運作
- 減少將任務與志工資料送到外部雲端模型的需求
- 降低 API 成本
- 在網路不穩時仍有機會維持本地推理

### 3.2 雙模型概念

MVP 可使用兩種模型角色：

| 模型角色 | 用途 |
|---|---|
| 派發模型 | 根據任務、技能、距離、緊急度產生派遣建議 |
| 偵錯模型 | 當派發模型輸出無效時，嘗試重新判讀或修正 |

### 3.3 本地演算法 fallback

災害情境不能假設 AI 永遠可用。因此 Dispatcher API 必須保留本地演算法 fallback。

fallback 可以採用：

```text
1. 過濾 availability = true 的志工
2. 依任務 urgency 從高到低排序
3. 優先考慮技能符合者
4. 計算 Haversine 距離
5. 選擇距離近且技能合適者
6. 回傳 ETA 與原因摘要
```

即使 Ollama 掛掉，系統仍可輸出基本派遣建議。

## 4. AI 使用邊界

本專案必須明確限制 AI 權限：

1. **AI 不發布撤離命令**  
   撤離、封路、醫療調度等正式命令必須由具權責單位決定。

2. **AI 不宣稱災害已確定發生**  
   沉默風險分數代表「需要確認」，不是「已發生災害」。

3. **AI 不編造資料**  
   AI 只能根據輸入 JSON、API 輸出與系統資料產生摘要，不得補充不存在的道路、傷亡、物資或感測器資訊。

4. **AI 必須可被人工審核**  
   每個 AI 輸出應包含原因摘要，讓指揮者知道建議從何而來。

5. **AI 失敗必須可降級**  
   若 AI 無法連線、輸出無法解析或延遲過高，系統應回到本地規則與演算法。

## 5. 建議 prompt 原則

如果需要讓 LLM 產生指揮摘要，建議使用強限制 prompt：

```text
你是防災應變指揮輔助助手。
請只根據提供的 JSON 資料產生建議。
不要宣稱災害已經確定發生。
不要發布正式撤離命令。
不要編造資料中沒有的道路、傷亡、感測器或通報資訊。
請輸出：
1. 優先確認區域
2. 原因
3. 建議現地查證動作
4. 建議通訊聯繫動作
5. 資料限制
```

## 6. 資料治理與隱私

Dispatcher API 應使用去識別化 ID，例如：

- `volunteer_id`
- `task_id`
- `incident_id`

除非必要，不應在 API 輸入中放入志工姓名、電話、身分證字號、住址等個資。若未來要接入真實志工系統，個資應由原系統保存，本元件只接收必要的去識別化任務資料。

## 7. 建議揭露方式

在 README 或簡報中，應坦白說明：

- 哪些部分使用 AI
- AI 的輸入資料是什麼
- AI 輸出如何被限制
- AI 失敗時如何 fallback
- 目前模型是否使用真實災害標籤
- 哪些結果只能作為建議，不能作為正式命令

這樣比單純宣稱「我們用了 AI」更有說服力，也比較符合防災場景對可靠性與責任邊界的要求。
