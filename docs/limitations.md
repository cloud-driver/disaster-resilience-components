# 限制、風險與改進方向

本文件用來主動揭露目前 MVP 的限制。這不是扣分項；相反地，清楚說明限制可以讓作品看起來更成熟、更可信。

## 1. 沉默災區偵測 API 的限制

### 1.1 通報資料目前可能是 mock data

若目前使用模擬通報資料，則只能驗證「高風險但低通報」的邏輯，不能宣稱已經能完整反映真實通報狀況。

改進方向：

- 串接 LINE Bot 通報資料
- 串接表單通報資料
- 串接地方政府災情查報系統
- 串接巡查回報結果

### 1.2 神經網路模型若使用 pseudo-label，不能宣稱具備真實預測能力

如果模型是用規則式分數產生 pseudo-label 再訓練，則它目前主要是在驗證 scoring layer 架構，而不是已經學到真實災害規律。

改進方向：

- 使用歷史災情紀錄重新訓練
- 使用巡查結果作為 label
- 使用通報延遲資料驗證「沉默」是否真的代表漏報
- 加入專家標註資料

### 1.3 即時資料來源可能不穩定

雨量、路況、土石流等資料來源可能受外部 API 可用性、格式改版、連線失敗或授權限制影響。

改進方向：

- 保存 raw snapshot
- 標記資料時間
- 設計 fallback 到上次成功資料
- 在 API 回應中揭露資料來源狀態

### 1.4 空間資料與村里邊界可能有版本差異

不同資料集的座標系統、邊界版本、更新時間可能不一致，可能影響空間 join 結果。

改進方向：

- 統一座標系統
- 紀錄資料版本
- 在 docs 中列出資料來源與更新日期
- 加入資料前處理檢查腳本

## 2. 救災志工智慧分配 API 的限制

### 2.1 AI 派遣建議可能不穩定

LLM 可能輸出格式不一致、無法正確引用志工 ID，或產生看似合理但不完全符合資料的建議。

改進方向：

- 加入嚴格 JSON output schema
- 使用 function calling 或 structured output
- 對 AI 輸出做 validation
- AI 失敗時立即 fallback 到本地演算法

### 2.2 目前本地演算法可能過於簡化

若 fallback 只看距離或簡單技能匹配，可能忽略：

- 志工負荷
- 過去派遣紀錄
- 道路是否可通行
- 志工裝備
- 任務需要多人合作
- 安全風險

改進方向：

- 加入任務所需人數
- 加入志工最大任務數
- 加入道路可達性
- 加入技能權重
- 加入任務完成後回報資料

### 2.3 志工個資與安全問題

派遣系統可能接觸志工位置、技能與出勤狀態。若資料處理不當，可能造成個資風險。

改進方向：

- 使用去識別化 ID
- 不在 API 中傳送姓名、電話、身分證字號或住址
- 限制 location 精度
- 加入權限控管
- 留存必要但不過量的日誌

### 2.4 Ollama 本地推理依賴硬體與模型

本地 LLM 的速度與穩定性會受硬體、模型大小、GPU、記憶體與首次載入時間影響。

改進方向：

- 提供輕量模型建議
- 允許關閉 AI，只使用本地演算法
- 設定 timeout
- 預熱模型
- 在 health endpoint 顯示 Ollama 狀態

## 3. 兩元件整合的限制

### 3.1 沉默風險不等於救援任務

沉默災區 API 的輸出代表「需要確認」，不應直接變成「派人進入危險區」。中間必須有人類指揮者或規則層判斷任務類型與安全性。

改進方向：

- 增加任務產生規則
- 將任務分為遠端聯繫、現地確認、道路確認、物資確認等不同等級
- 高風險區域先建議聯絡村里長或消防分隊，不一定立即派志工進入

### 3.2 GeoJSON 座標轉任務座標需要小心

使用村里中心點可能不代表真正風險位置。某些村里面積很大，中心點可能在不適合派遣的位置。

改進方向：

- 使用風險熱點座標
- 使用道路可達點
- 使用避難所、村里辦公室或消防分隊作為集合點
- 在任務中標記座標來源

### 3.3 評分結果需要可解釋

如果只給分數，指揮者很難信任系統。必須提供 `silent_reason`、資料時間、資料來源與分數拆解。

改進方向：

- 顯示規則式分數與模型分數
- 顯示報告數、雨量、路況、感測器覆蓋缺口
- 在地圖上呈現原因標籤

## 4. 不應該做的宣稱

為了避免過度承諾，簡報與 README 中不應出現以下說法：

| 不建議說法 | 建議改法 |
|---|---|
| 我們可以準確預測災害 | 我們提供高風險低通報區域的優先確認清單 |
| AI 會自動決定救災派遣 | AI 提供派遣建議，最後由人類指揮者審核 |
| 沒有通報就代表一定有問題 | 沒有通報但風險高，代表需要進一步確認 |
| 系統可以直接取代政府平台 | 本作品是可拼接的防災元件，可整合到既有系統 |
| 模型已經能正式上線 | MVP 已能驗證資料流與核心功能，正式部署前需更多資料與驗證 |

## 5. Roadmap

### 短期

- 補齊主 repo 的 bilingual README 與 docs
- 加入整合 demo script
- 補上 sample input / output
- 確認兩個 API 的 Swagger 都能打開
- 加入 task generator 範例

### 中期

- 串接真實通報資料
- 加入道路可達性判斷
- 強化志工技能權重與任務人數需求
- 將 AI 輸出改為 structured JSON
- 加入 Docker Compose demo

### 長期

- 使用歷史災情與巡查結果重新訓練模型
- 與 LINE Bot、Google Chat 或 Web Dashboard 串接
- 建立人類審核介面
- 建立任務回報回流機制
- 發展成可供其他公民科技團隊重複使用的防災元件庫
