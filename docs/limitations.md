# MVP 限制、風險與後續方向

## 已完成的可驗證範圍

- 沉默元件可產生以花蓮縣村里為單位的規則式沉默風險結果。
- 具備 LINE 回報、`pending → verified / rejected` 審核與已驗證事件 queue 的設計。
- 可輸出 JSON、CSV、GeoJSON 與資料可信度 metadata。
- 志工元件可依技能、距離、急迫度與可用性產生 deterministic 派工建議。
- 母 repo 的離線 sample demo 可驗證任務轉換、候選推薦與重複指派避免。
- 有 `IntegratedTask` JSON Schema 與目標 OpenAPI 契約。

## 不能誇大的地方

| 項目 | 真實限制 |
|---|---|
| 災害預測 | 沉默分數不是災害發生機率；NN 實驗層也沒有真實 ground truth。 |
| 即時性 | 外部 API、網路與 pipeline 執行時間會影響資料可用性。 |
| 水文資料 | 水利署水位資料尚未完成整合。 |
| 路況影響 | 目前仍以規則式特徵為主，空間影響範圍可再細化。 |
| LINE 對外使用 | 需公開 HTTPS 網域、LINE Developers 設定與營運管理。 |
| `refresh=true` | 為同步阻塞操作，不適合開放給不受信任使用者。 |
| 志工持久化 | 活動設定與報名資料目前為記憶體保存，重啟後消失。 |
| 端對端整合 | 母 repo demo 沒有呼叫兩個實際 HTTP 服務；adapter／facade 尚未落地。 |
| 外部授權 | 志工元件的完整對外身份驗證與管理授權尚需補強。 |
| 地理與技能資料 | 志工位置、技能與可用性若過期，推薦品質會下降。 |

## 優先後續工作

### P0：安全與誠實性

1. 把所有 demo／sample／batch／live 狀態放入 UI 與 response 顯著位置。
2. 建立正式 adapter，驗證 schema 並保存完整轉換／派工 audit log。
3. 將志工服務改為持久化資料庫，加入管理者權限與操作紀錄。
4. 將兩服務放到 HTTPS 反向代理後，限制管理 endpoint 與 refresh 操作。
5. 建立個資告知、保存期限、撤回與刪除流程。

### P1：可靠性與可部署性

1. 沉默元件改用排程／背景 job 取代公開同步 refresh。
2. 補 Docker、版本鎖定、健康檢查與 CI。
3. 建立重試、資料來源降級與 alert 機制。
4. 針對志工身分、位置、技能和 availability 建立更新與到期策略。
5. 建立任務管理與人工審核 UI。

### P2：資料與研究品質

1. 納入更多水文、道路、通訊與地方巡查資料。
2. 使用歷史災情與現地確認記錄建立真實標註資料。
3. 建立模型校準、偏誤檢查、交叉驗證與持續監測。
4. 評估不同地區、不同災種與不同人口特徵下的排序穩定性。
5. 讓任務類型與技能 ontology 可配置、可版本化。

## 應持續維持的底線

- 沒有真實標註與實測，不談預測準確。
- 沒有來源可信度與人工審核，不把資料轉成行動命令。
- 沒有正式 adapter／audit，不宣稱完成端對端自動整合。
- 沒有資安與個資治理，不進入實際大規模人員調度。
